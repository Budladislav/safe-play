# Safe Play — правила рабочего процесса

Эти правила действуют для каждого следующего патча проекта, если пользователь явно не попросил иначе.

1. Работать прямо в локальной ветке `main`, без создания дополнительных веток.
2. Делить реализацию патча на логические коммиты.
3. Во время разработки добавлять изменения в раздел `Unreleased` файла `CHANGELOG.md`.
4. После полной реализации выполнить все релевантные автоматические тесты, проверки синтаксиса и визуальную проверку интерфейса.
5. Затем оформить отдельный номерной релиз: выбрать версию по масштабу изменений, обновить её и дату релиза во всех релевантных местах, перенести записи из `Unreleased` в секцию версии и создать отдельный release-коммит.
6. Не выполнять `git push` без отдельной явной команды пользователя после его тестирования.
7. После готовности оставить локальный dev-сервер запущенным и сообщить адрес для теста.
8. Не изменять и не удалять пользовательские данные во время проверки; для изменяющих сценариев создавать и затем удалять только явно тестовые записи.
